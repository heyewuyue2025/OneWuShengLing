## 三、踩坑记录

### 1. LPI 单位不一致
**问题**：同一数据集中 `Units` 有 Individuals、breeding pairs、mean counts per route 等，直接跨物种求和无意义。  
**解决**：以各物种时间序列自身起始值为基准（=100）做指数化，再取中位数聚合。  
**教训**：宏观指标的定义比画图更重要，需在叙事中标注方法论。

### 2. 大文件处理超时
**问题**：LPD_2024_public.csv 约 3.6 万行，浏览器端直接加载 CSV 会卡顿，全目录 Glob 扫描超时。  
**解决**：Python 预处理为轻量 JSON（lpi.json 约 120KB），网页离线可演示。  
**教训**：比赛现场时间紧，预烘焙数据是正确取舍。

### 3. 濒危种群数量文本化
**问题**：`Estimated Population` 为 "Unknown"、"< 500"、"5 mature individuals" 等自由文本。  
**解决**：正则提取数字 + qualifier 标记（less_than / unknown），排序时 unknown 置后。  
**教训**：情感叙事可用 raw 文本展示，统计分析需结构化解析并标注不确定性。

### 4. PowerShell 路径编码
**问题**：中文路径在 PowerShell 脚本中显示乱码，`&&` 运算符不支持。  
**解决**：使用 `Set-Location` + 分号分隔命令，Python 脚本内用 `Path(__file__)` 相对定位。

## 四、技术取舍决策

### 为何选 Scrollytelling 而非 Dashboard？
- 比赛核心命题是「叙事」，Dashboard 适合探索但缺乏故事节奏
- Scrollytelling 能自然串联四组异构数据集（LPI / 濒危 / 动物档案 / 企鹅）
- 滚动驱动的图表切换降低认知负担，评委演示时体验更连贯

### 为何用预烘焙 JSON 而非实时 CSV？
- LPI 数据量大，现场 8 小时需留足叙事打磨时间
- JSON 可离线演示（外滩现场网络不确定）
- 预处理阶段完成数据校验，减少前端 bug 风险

### 为何用 Intersection Observer 而非 Scrollama/GSAP？
- 零依赖，减少 CDN 失败风险
- 核心逻辑 30 行，便于现场快速调试
- 满足 `-40%` rootMargin 的「屏幕中央触发」效果

## 五、可复用的创作策略

1. **先找跨数据集的「对照句」**：如「宏观在跌，微观在喘」——用一句话串联 LPI 与企鹅健康数据
2. **情感锚点用最小数字**：濒危物种卡片展示种群 ≤10 的个案，比百分比更有冲击力
3. **方法论透明**：尾声注明 LPI 指数计算方式，增强可信度（澎湃美数课风格）
4. **AI 做体力活，人做审美与逻辑**：清洗脚本和 D3 模板交给 AI，叙事节奏和色彩由人终审
5. **边做边记**：踩坑立即写入本文件，避免赛后回忆遗漏
